Так причем тут АОП?

Выделение сквозной функциональности в модули.

В концепции АОП фигурируют 7 понятий. Давайте их рассмотрим.

  • аспект (aspect) – модуль или класс, реализующий сквозную функциональность. Аспект изменяет поведение остального кода, применяя совет в точках соединения, определённых некоторым срезом. Так же аспект может использоваться для внедрения функциональности;
  • совет (advice) – дополнительная логика — код, который должен быть вызван из точки соединения. Совет может быть выполнен до, после или вместо точки соединения;
  • точка соединения (join point) — точка в выполняемой программе (вызов метода, создание объекта, обращение к переменной), где следует применить совет ;
  • срез (pointcut) — набор точек соединения. Срез определяет, подходит ли данная точка соединения к заданному совету;
  • внедрение (introduction) — изменение структуры класса и/или изменение иерархии наследования для добавления функциональности аспекта в инородный код;
  • цель (target) – объект, к которому будут применяться советы;
  • переплетение (weaving) – связывание объектов с соответствующими аспектами (возможно на этапе компиляции, загрузки или выполнения программы).

Можно вытянуть по ним информацию, без разбора файла как кода, но как программы.

Например сторонняя библиотека со своими аттрибутами может не изменять поведение класса, а просто использовать его для своих нужд, например генерации документации

  • Swagger
  • Библиотека предоставляет набор декораторов и/или базовых классов
  • У нас есть класс с бизнесс логикой
  • Декорируем класс
  • Декоратор применяется на этапе инициализации и возвращает декорированный класс
  • Создаем инстанс этого класса
    • Проксирующий класс-обертка создает инстанс класса
    • Создается проксируемый объект
    • Записывается в хранилище - { инстанс обертки: проксируемый класс }
  • Обращение с проксируемым объектом происходит через class API
  • А что GC? Следит за прокси-инстансом и при его уничтожении не очистится проксируемый объект
  • Чтобы избежать утечек памяти используем слабое хранилище
  • Клиентский код обращается к члену классу
  • Член класса проксирует запрос
  • В классе есть ссылка на хранилище
  • Находит ссылку на соответствующий объект в хранилище
  • Вызывает необходимый метод на хранилище
  • Тем самым проксирует запрос

Рассмотрим способ организации продвинутой схемы с хранилищем.

  1. АОП Библиотека предоставляет нам свои функции, классы, декораторы.
  2. В клиентском коде мы имеем свои классы, реализующие бизнесс логику.
  3. Декорируем существующие классы. "Помечаем" члены или наследуемся от базового библиотечного класса.
  4. Декороры записывают информацию в shared хранилище, которое находится внутри пакета.
  5. И возвращает прозрачный проксирующий класс-обертку с доопределенным конструктором.
  6. Обращаясь к классу снаружи
  7. При создании экземпляра класса
  8. Доопределенный конструктор просканирует "помеченные" члены
  9. Вытянет метаинформацию о них и доопределит их поведение в соответствии с реализуемой сквозной функциональностью.
  10. Обращаясь к инстансу
  11. Получаем доопределенный сквозной эффект
